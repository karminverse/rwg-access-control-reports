# RWG — Access Control Reports

The public directory for the **Inverse Finance Risk Working Group** access-control reports:
per-asset access control assessments of assets under review for FiRM collateral, plus weekly
Collateral Integrity Checkups. Deployed on Vercel; the landing page auto-indexes whatever report
files are present and self-enriches each card from the report's own header.

> The reports themselves are generated by **Trustfall** (the RWG's access-control scanner) and the
> weekly checkup skill, which live in separate workspaces. This repository is just the publishing
> surface: drop the output files in, push, and they appear.

Each report traces the full privilege graph (roles, holders, dependency contracts) and tracks the
critical parameter levers, mint caps, debt ceilings, fee rates, and oracle pointers, that can move
risk in a single transaction.

## Deploy to Vercel

This deploys with **zero special configuration** — `vercel.json` already tells Vercel everything it
needs. No env vars, no build command, no framework.

1. Push this folder to its own (private) GitHub repo:
   ```bash
   git init
   git add .
   git commit -m "RWG access control report directory"
   git remote add origin <your-new-github-repo>
   git push -u origin main
   ```
2. In Vercel: **Add New → Project → Import** that repo.
3. Leave every setting at its default and click **Deploy**:
   - **Framework Preset:** Other (auto-detected — there is no framework)
   - **Root Directory:** `./` (the repo root)
   - **Build Command / Output Directory:** leave empty (there is no build step)
   - **Environment Variables:** none

That is the entire setup. On every deploy Vercel automatically:
- serves all static files (`index.html`, `viewer.html`, the vendored `marked.min.js` / `purify.min.js`, and everything under `reports/`), and
- turns `api/reports.js` into a serverless function at `/api/reports`, bundling `reports/**` (via
  `vercel.json` `includeFiles`) so the function can list and read the reports at request time.

**Private repo, public page (Vercel Pro):** supported out of the box — the repo stays private while the
production URL is public. If you have ever enabled **Settings → Deployment Protection → Vercel
Authentication**, confirm it is **off for Production** (it is off by default) so anyone can view the page.

**Custom domain (optional):** Project → Settings → Domains → add your domain.

## Adding / updating reports (the everyday flow)

No Vercel dashboard steps, ever — and nothing to regenerate. Just commit and push:

1. Drop the files into the repo:
   - collateral asset → `reports/collateral/<Asset>.html` **and** `reports/collateral/<Asset>.md`
   - weekly checkup → `reports/weekly/<YYYY-MM-DD>.md`
2. *(collateral only, optional)* add a prose entry to `reports/manifest.json` (protocol, summary,
   `status`, display-name overrides, `order`). A report with no entry still appears, named from its header.
3. `git add . && git commit -m "add <asset>" && git push`

Vercel redeploys on the push and the new card appears. The function re-indexes `reports/` live, so the
factual card stats (contracts, privileged fns, EOA holders, critical roles, upgradability, scan date)
populate automatically from the report's own Surface Summary header.

## Layout

```
.
├── index.html              # Landing directory (3 sections; no build step)
├── viewer.html             # Dark-theme Markdown renderer (weekly checkups + .md reports)
├── favicon.svg             # Site icon (inline SVG; no binary, CSP-safe)
├── api/reports.js          # Serverless function: indexes reports/, parses each report header, merges the manifest
├── marked.min.js           # Vendored Markdown renderer for viewer.html (no CDN dependency)
├── purify.min.js           # Vendored DOMPurify — sanitizes the viewer's rendered HTML
├── package.json            # Pins the Node runtime
├── vercel.json             # Bundles reports/** into the function
└── reports/
    ├── manifest.json       # Optional per-report PROSE overlay (protocol, summary, status, order)
    ├── collateral/         # FiRM collateral assessments  →  drop <Asset>.html + <Asset>.md
    └── weekly/             # Weekly checkups               →  drop YYYY-MM-DD.md
```

There is **no build step** for Vercel. The page calls `/api/reports` at load time; that function lists
the folders, pairs each HTML report with its Markdown sibling, reads the **Surface Summary** and
metadata out of each report's own Markdown header, parses each weekly checkup's header for a summary,
and overlays `reports/manifest.json`.

## `reports/manifest.json` schema

Optional prose overlay, keyed by file **basename** (no extension). Every field is optional, and the
whole file is optional. All factual stats come from the reports themselves, so the manifest only
carries what a report does not contain.

```jsonc
{
  "collateral": {
    "crvUSD": {
      "asset":    "crvUSD",            // display name override (default: token symbol from the report)
      "subtitle": "crvUSD Stablecoin", // small line under the name (default: contract name from the report)
      "protocol": "Curve Finance",
      "summary":  "One-line description of the asset.",
      "status":   "reviewed",          // reviewed | in-review | monitoring | parked
      "order":    2                    // sort order (ascending; unset → alphabetical)
    }
  },
  "weekly": {}
}
```

**Honesty of the card.** `status` is the analyst's review position. Everything else on a card
(contracts, privileged fns, EOA holders, critical roles, upgradability, scan date) is read straight
from the published report and is shown as a neutral factual stat, **not** an overall risk grade.

## Security

A read-only publishing surface with a deliberately small attack surface:

- **No secrets, no auth, no write surface.** Static files plus one **read-only** directory-lister
  function (`/api/reports`). The function takes no user input — it lists fixed folders — so there is no
  path-traversal or injection vector through it.
- **Output is escaped / sanitized.** The directory page HTML-escapes every dynamic value; the Markdown
  viewer renders through **DOMPurify** (`purify.min.js`) so report content cannot inject active markup.
- **No third-party resources at runtime.** `marked` and `DOMPurify` are vendored locally and pinned —
  the deployed pages load nothing from external origins.
- **Viewer path guard.** `viewer.html` only opens `/reports/**.md` (decoded-path validated; traversal,
  null bytes, and off-site URLs rejected; same-origin fetch only).
- **Security headers** (`vercel.json`): `Content-Security-Policy`, `X-Content-Type-Options: nosniff`,
  `X-Frame-Options: SAMEORIGIN`, `Referrer-Policy`. The CSP blocks all external origins; it permits
  `'unsafe-inline'` for scripts because the directory page and the scanner-generated report HTML use
  inline scripts — external script injection is still blocked, and inline-XSS is already neutralized by
  the escaping and DOMPurify above.

Nothing here needs per-report maintenance — these protections apply automatically to any report dropped in.
